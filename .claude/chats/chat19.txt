---- 

1.Docker 配置和 OCI runtime 配置里，给这个实例写的 /data 源其实是正确的。
都是宿主那份 image-backed 的持久路径，也就是 /data/android_data/实例ID。

2.但运行中的容器并没有实际用到这份 image-backed 路径。
容器内 /data 的 live mountinfo 显示，它实际绑到的是宿主 /data 分区底层上的一个普通子目录视图，设备号是 nvme 分区，不是 loop17。

3.当前宿主侧这个实例的持久数据还在 loop17 上。
而当前运行态容器的 /data 却来自 nvme 根分区里的另一套目录。
这就是为什么：
宿主还能看到旧 APK、旧 packages.list、旧 uiderrors；
但运行中的实例里 /data/app 为空，packages.xml 和 uiderrors.txt 都在 08:31 重新生成。

4.所以这里更像是一个挂载时序或挂载命名空间问题：
容器创建 / 重启时，/data/android_data/实例ID 这条路径还没有指向 loop 镜像，
或者容器创建所在的命名空间没有看到那条 loop 挂载，
结果 bind mount 抓到了“底层被覆盖前目录”。
后面宿主再把镜像挂到同一路径上，只会覆盖宿主视图，不会改变已经进入容器的旧 bind 结果。

机制上已经露头了：宿主的 /data/android_data/<实例> 是 shared 的 loop 挂载点，而我们之前从 Docker 配置看到容器 /data 的 bind Propagation:rprivate。这正好符合“容器先 bind 到底层空目录，宿主随后把 loop 镜像挂上去，但新 overmount 不再传播进容器”的经典分叉模型。下面我补两块直接证据：当前容器实际 mountinfo，以及负责 mount loop + docker start 的宿主实现路径。

----

补上直接证据后，这条“视图分叉”链路现在基本已经闭环：

1. Docker 静态配置写的是宿主路径 `/data/android_data/<实例>`，但 propagation 明确是 `rprivate`。

对 74ae 实例现场抓到的 `docker inspect` 关键字段：

- `Source: /data/android_data/RK8052B1202500035_74ae20fe-498f-4093-b358-377ef6713edd`
- `Destination: /data`
- `Propagation: rprivate`

2. 宿主当前这条实例路径已经是单独的 loop mount，而且不是底层 `/data` 本体，而是覆盖在其上的另一层 mount 对象。

宿主 `mountinfo`：

- `1289 211 7:17 / /data/android_data/RK8052B1202500035_74ae20fe-498f-4093-b358-377ef6713edd rw,relatime shared:652 - ext4 /dev/loop17`
- 父挂载仍是：`211 24 259:6 / /data rw,relatime shared:106 - ext4 /dev/nvme0n1p6`

也就是说，宿主肉眼看到的 `/data/android_data/<实例>`，此时已经是 `/dev/loop17` 提供的镜像视图。

3. 但 live 容器里的 `/data` 并没有指到这个 `/dev/loop17`，而是仍然钉在底层 nvme 分区的子目录对象上。

live 容器 `mountinfo`：

- `1495 1377 259:6 /android_data/RK8052B1202500035_74ae20fe-498f-4093-b358-377ef6713edd /data rw,relatime shared:849 - ext4 /dev/nvme0n1p6`

这行最关键的事实不是路径字符串，而是设备号和源挂载对象：

- 宿主正确数据面是 `7:17 /dev/loop17`
- live 容器 `/data` 却是 `259:6 /dev/nvme0n1p6`

所以当前不是“同一挂载在宿主和容器里看起来不同”，而是容器 `/data` 从一开始就落在了底层 `/data` 分区的旧 mount 对象上，没有跟上宿主后续 overmount 出来的 loop 镜像。

4. 宿主恢复链的实现面也已基本定位到 IaaS agent 本身，而不是 Docker 配置文件。

现场进程树显示，这批恢复 agent 都是在宿主 reboot 后同一秒被重新拉起：

- `08:30:18 /data/bin/AC_IaaS_agent`
- `08:30:18 /data/bin/Virtual_Host_Agent`
- `08:30:18 /data/bin/Virtual_Native_Agent`
- `08:30:18 /data/bin/iaas_agent_keepalive`

它们的父进程链是：

- `/usr/lib/systemd/systemd` -> `runsv native_host_d` -> `Virtual_Host_Agent`
- `/usr/lib/systemd/systemd` -> `runsv native_agent_d` -> `Virtual_Native_Agent`
- `/usr/lib/systemd/systemd` -> `runsv native_iaas_d` -> `AC_IaaS_agent`

也就是说，宿主 reboot 后是 runsv 管着的 native agent 套件在恢复实例，而不是某个临时手工脚本。

5. 更关键的是，agent 二进制字符串里直接命中了我们需要的那条链：

`/data/bin/AC_IaaS_agent` 内直接出现：

- `docker start %s`
- `/data/DockerMount/`
- `tag:yaml.org,2002:/data/android_data`
- `mount -o loop %s %s`
- `/data/android_data/%s`

`/data/bin/Virtual_Host_Agent` 内直接出现：

- `docker inspect`
- `docker start %s`
- `docker restart`
- `/data/DockerMount/`
- `/data/android_data/`
- `mkdir -p /data/android_data/`
- `mount -o loop %s%s.img /data/android_data/%s`
- `docker create -it --restart=always --name=%s ... -v /data/android_data/%s:/data ...`

这说明“实例镜像文件 `/data/DockerMount/*.img` -> mount 到 `/data/android_data/<实例>` -> 再把这个路径 bind 给容器 `/data`”这套逻辑，本身就实现在宿主 IaaS agent 里。

因此现在可以把问题机制明确成一句更硬的表述：

宿主 08:30 reboot 之后，IaaS 恢复链重新拉实例时，容器 `/data` 对 `/data/android_data/<实例>` 的 bind 抓到的是底层 `/data` 分区上的旧目录对象；之后宿主侧又把 `/data/DockerMount/<实例>.img` loop mount 到同一路径上，但 live 容器没有切到新的 loop 挂载对象，于是宿主视图和容器视图永久分叉。

这里最像的具体失配方式就是：

- 恢复顺序是“先 create/start 容器，再完成 loop overmount”，或者
- loop mount 虽然已完成，但完成在宿主另一个 mount 对象/传播域里，容器创建时继承到的不是最终那个 mount

无论是哪一种，核心都不是 APK 被卸载，而是实例恢复时 `/data` bind 指向了错误 mount 对象。

6. 还有两个辅助事实也支持这个判断：

- `/data/container_persistent_data/<实例>/mounts` 和 `mountstats` 仍停留在 `2026-05-07 19:48`，没有反映 5 月 11 日这次 reboot 后的错误 live `/data` 视图
- `go_agent.log` 与 2026-05-11 的归档日志里，没有直接打出针对 74ae 的 `mount -o loop` / `docker start` 细节，所以当前实现面证据主要来自 live mountinfo、进程树和 agent 二进制字符串，而不是操作日志正文

所以到这一步，调查重点已经可以从“是否卸载过应用”完全切换成：

- 为什么 IaaS 恢复链在 reboot 后把 bind `/data` 固定到了底层目录对象
- `AC_IaaS_agent` / `Virtual_Host_Agent` 在恢复实例时，究竟是先做 `docker create/start` 还是先做 `mount -o loop`
- 这条链路是否缺少 `mount --make-shared` / 正确 propagation，或在 runsv 多 agent 并发恢复时存在时序竞争

换句话说，当前最硬结论已经不是“reboot 后挂错数据面”这么宽泛，而是：

宿主 reboot 后，IaaS agent 恢复实例时对 `/data/android_data/<实例>` 这条宿主路径发生了 bind-before-final-overmount 的时序/传播失配，导致 live 容器 `/data` 固定在底层 `nvme0n1p6` 视图，而宿主后来看到的却是 `loop17` 镜像视图。

----

继续把启动链补到落盘目录级别后，可以再收紧一层：

1. `runsv` 不是泛泛的父进程，而是明确由 `runsvdir /etc/service` 管理。

现场进程链补充如下：

- `830 1 ... runsvdir /etc/service`
- `831 830 ... runsv native_host_d`
- `832 830 ... runsv native_agent_d`
- `834 830 ... runsv native_iaas_d`

并且 live `/proc` 直接证明这三个 `runsv` 的工作目录分别是：

- `/etc/service/native_host_d`
- `/etc/service/native_agent_d`
- `/etc/service/native_iaas_d`

`/proc/<pid>/fd` 里还能看到它们各自打开的 `supervise/lock` 和 `supervise/control`，说明这确实是标准 runit/runsv service 目录，不是名字相像的普通进程。

2. 宿主上确实存在对应的 service 定义，且 `run` 脚本非常薄，没有额外 shell 逻辑。

已确认存在：

- `/etc/service/native_host_d/run`
- `/etc/service/native_agent_d/run`
- `/etc/service/native_iaas_d/run`

脚本内容分别是：

- `native_host_d/run` -> `exec /data/bin/Virtual_Host_Agent >/dev/null 2>&1`
- `native_agent_d/run` -> `exec /data/bin/Virtual_Native_Agent >/dev/null 2>&1`
- `native_iaas_d/run` -> `exec /data/bin/AC_IaaS_agent >/dev/null 2>&1`

这点很关键，因为它排除了“恢复动作藏在 run 脚本里”的可能：

- `runsv` 层只负责拉起 agent
- 真正的实例恢复、`loop mount`、`docker create/start` 逻辑仍然在 agent 二进制本体里

因此当前对恢复顺序的判断边界也更清楚了：

- 启动链路已经坐实到 `/etc/service/*/run -> /data/bin/*Agent`
- 但 `mount -o loop` 与 `docker create/start` 的先后，仍需要继续从 `AC_IaaS_agent` / `Virtual_Host_Agent` 二进制内部或更细粒度运行时取证里拿直证

换句话说，启动层已经闭环，剩下未闭环的只剩“agent 内部恢复时序”这一小段。

----

继续往 agent 二进制内部收缩后，又拿到一层对“恢复顺序”很有用的新事实：

1. 宿主这两个关键 agent 都是未 strip 的 Go 可执行文件，而且带 `debug_info`。

现场拉回本地后的 `file` 结果：

- `AC_IaaS_agent`: `ELF 64-bit ... Go BuildID=... with debug_info, not stripped`
- `Virtual_Host_Agent`: `ELF 64-bit ... Go BuildID=... with debug_info, not stripped`

这意味着后续如果要继续追时序，不必盲猜汇编，直接可以沿 Go 符号和 `objdump` 调用图往下钻。

2. `AC_IaaS_agent` 的符号表确认它同时暴露了“恢复”和“挂载”两个入口：

- `IaasAgentGo/internal/app/container/controller/instanceLockChan.InstanceRestore`
- `IaasAgentGo/internal/app/container/controller/instanceLockChan.ImageMount`
- `IaasAgentGo/internal/app/container/controller/instanceLockChan.ContainerFactoryRestore`
- `IaasAgentGo/internal/app/container/service/containerFunction.Restore`

但从当前这几个入口的 `CALL` 级别 `objdump` 看，`InstanceRestore` / `ImageMount` 都先走进统一的：

- `IaasAgentGo/internal/app/container.HandleCommonController`

随后主要是接口分发调用，说明 AC_IaaS_agent 这边恢复逻辑被封在 common controller / 运行时分发表里，单靠入口函数本身暂时还看不到更细顺序。

3. `Virtual_Host_Agent` 这边则已经出现了更清晰的顺序骨架。

符号表直接暴露：

- `VirtualHostAgent/virtualization.CreateContainer`
- `VirtualHostAgent/virtualization.StartCon`
- `VirtualHostAgent/virtualization.setStorage`
- `VirtualHostAgent/virtualization.delImageStorageFile`

而 `CreateContainer` 的 `CALL` 顺序里，已经能明确看到：

- 前段调用了 `VirtualHostAgent/virtualization.setStorage`
- 之后才继续一串 `VirtualHostAgent/utils.LocalCommand` / `LocalCommandSU`
- 另有单独的 `VirtualHostAgent/virtualization.StartCon` 负责 start 路径

这至少说明在 Host_Agent 里，“存储准备”和“容器 start”不是同一个黑盒动作，而是被拆成不同阶段 / 不同函数。

4. `setStorage` 自身的调用图也支持它就是存储侧准备入口，而不是普通配置函数。

`setStorage` 内部已看到：

- `VirtualHostAgent/utils.PathExists`
- `os.MkdirAll`
- 多次 `fmt.Sprintf`
- 多次 `VirtualHostAgent/utils.LocalCommand`

这类组合很像：

- 检查并创建 `/data/android_data/<实例>` 一类宿主路径
- 拼接 shell 命令
- 执行本地命令完成存储准备 / mount 相关动作

同时 `CreateContainer` 在调用 `setStorage` 失败时，还会进入：

- `VirtualHostAgent/virtualization.delImageStorageFile`

这进一步说明 `setStorage` 处理的就是和镜像 / 挂载文件直接相关的前置步骤。

因此，虽然现在还没有把 `mount -o loop` 那一条 shell 命令在反汇编里逐字抓出来，但恢复顺序已经可以比之前再收紧一层：

- `Virtual_Host_Agent` 的 create 路径里，先有一个明确独立的存储准备阶段 `setStorage`
- 后面才继续执行创建/启动容器所需的本地命令
- `StartCon` 是单独存在的 start 路径，而不是和存储准备揉在一起

所以当前更像的机制已经从“可能是先 bind 再 overmount”进一步收缩成：

宿主恢复链里至少存在一个先做 storage setup、后做 container create/start 的显式分段；如果这两个阶段之间的 mount 对象 / propagation 没处理对，就足以造成当前看到的 `/data` 视图分叉。

5. 当前还缺的最后一跳直证是：

- `setStorage` 里具体拼出来的 shell 命令是否就是 `mount -o loop ... /data/android_data/<实例>`
- `CreateContainer` 后半段具体拼出来的命令是否对应 `docker create -v /data/android_data/<实例>:/data`
- `StartCon` 是否只负责 `docker start <实例>`

但即使这三句命令正文还没完全抠出来，恢复时序的控制边界已经比之前清楚很多：

- 存储准备在 `Virtual_Host_Agent` 的 `setStorage`
- 容器创建在 `Virtual_Host_Agent` 的 `CreateContainer`
- 容器启动在 `Virtual_Host_Agent` 的 `StartCon`
- 更高层的恢复分派在 `AC_IaaS_agent` 的 `InstanceRestore` / `ImageMount` / `HandleCommonController`

换句话说，剩下要补的不是“谁负责恢复”这种大方向，而只是这些已定位函数内部的命令级顺序细节。

----

继续做了一次最窄验证后，命令模板和函数名已经基本能直接对上：

1. `Virtual_Host_Agent` 的符号表精确命中三段关键函数：

- `VirtualHostAgent/virtualization.setStorage`
- `VirtualHostAgent/virtualization.CreateContainer`
- `VirtualHostAgent/virtualization.StartCon`

2. 同一个二进制里直接存在三类关键命令模板原文：

- `mount -o loop %s%s.img /data/android_data/%s`
- `docker create -it --restart=always --name=%s ... -v /data/android_data/%s:/data ...`
- `docker start %s`

而且这三类字符串都在 `Virtual_Host_Agent` 本地样本里现场重新验证到了，不再只是先前远端 `strings` 的一次命中。

3. 把这组字符串和前面已经拿到的函数/调用图合在一起后，当前最合理、也最硬的函数职责划分是：

- `setStorage`：负责镜像/宿主目录准备，以及 `mount -o loop ... /data/android_data/<实例>` 这一侧的动作
- `CreateContainer`：负责拼 `docker create ... -v /data/android_data/<实例>:/data ...`
- `StartCon`：负责 `docker start <实例>`

也就是说，恢复链里至少已经能明确拆成三段：

- 先准备 storage 并挂到 `/data/android_data/<实例>`
- 再用这个宿主路径去 `docker create` 绑定容器 `/data`
- 最后单独 `docker start`

4. 这让“问题出在哪一层”进一步收敛：

既然 `mount -o loop`、`docker create -v /data/android_data/...:/data`、`docker start` 都在同一个 `Virtual_Host_Agent` 里按阶段出现，那么当前异常更像不是 Docker 自己的随机行为，而是 Host_Agent 这条恢复链在 storage setup 与 container create/start 交界处，对 mount 对象/传播域的处理出了时序问题。

换句话说，到现在已经可以把怀疑点非常具体地压缩到：

- `Virtual_Host_Agent.setStorage`
- `Virtual_Host_Agent.CreateContainer`
- `Virtual_Host_Agent.StartCon`

这三个函数内部的命令执行顺序与挂载可见性，基本就是后续再追的唯一主战场。

----

继续把 `Virtual_Host_Agent` 这三个函数直接反编译后，恢复顺序现在已经不是推断，而是伪代码级别的直证：

1. `setStorage` 的命令顺序已经完整展开。

反编译直接显示它按如下顺序执行本地命令：

- `dd if=/dev/urandom of=%s%s.img bs=1M count=0 seek=%d`
- `losetup -f`
- `losetup %s %s%s.img`
- `mkfs.ext4 <loop设备>`
- `losetup -d <loop设备>`
- `mount -o loop %s%s.img /data/android_data/%s`

也就是说，`setStorage` 的确不是泛化的“存储准备”，而是非常具体地：

- 创建实例镜像文件
- 借 loop 设备格式化 ext4
- 最终把镜像直接 mount 到 `/data/android_data/<实例>`

2. `CreateContainer` 里能直接看到调用顺序：

- 先检查容器名是否已存在：`docker ps -a --format '{{.Names}}' |grep -Fx <name>`
- 检查镜像是否存在：`docker images --format ... | grep ...`
- 准备网络与 `/data/android_data/<实例>` 目录
- 调 `getCount(...)`
- 调 `setStorage(name, storage, ...)`
- `setStorage` 失败时立刻进入 `delImageStorageFile(name, ...)` 做回滚
- `setStorage` 成功后，才 `fmt.Sprintf(...)` 出完整的 `docker create -it --restart=always --name=%s ... -v /data/android_data/%s:/data ...` 命令
- 然后通过 `VirtualHostAgent_utils_LocalCommand(...)` 真正执行 `docker create`

这一步已经把关键时序钉死了：

- `mount -o loop ... /data/android_data/<实例>` 在 `setStorage` 里
- `docker create -v /data/android_data/<实例>:/data ...` 在 `CreateContainer` 后半段
- 两者之间不是并列猜测关系，而是明确的先后调用关系：`CreateContainer -> setStorage -> docker create`

3. 更关键的是，`docker start` 也不是另一个模糊路径，而是在 `CreateContainer` 里按条件直接跟在 `docker create` 后面。

反编译显示：

- `CreateContainer` 成功执行 `docker create` 后
- 若 `*v219->Data.IsStart` 为真
- 它就在同一个函数里继续 `fmt.Sprintf("docker start %s", name)`
- 再走一次 `VirtualHostAgent_utils_LocalCommand(...)`

所以就当前这版 Host_Agent 而言，最准确的顺序不是抽象的 `setStorage -> CreateContainer -> StartCon`，而是：

- `CreateContainer` 内部先调用 `setStorage`
- `CreateContainer` 自己执行 `docker create ... -v /data/android_data/<实例>:/data ...`
- 若 `IsStart` 为真，`CreateContainer` 继续在本函数后半段直接执行 `docker start <实例>`

4. `StartCon` 独立函数也存在，但它更多像是单独的“对已有容器执行 start”的接口，不是宿主恢复新建路径里唯一的 start 入口。

它自己的伪代码也很清楚：

- 先跑 `docker inspect --format='{{.State.Running}}-{{.State.Status}}' <name>`
- 若不是已运行态
- 再拼 `docker start <name>` 并执行

这说明：

- `StartCon` 是“已有容器单独 start”的公共接口
- 但在实例恢复/新建这条主路径里，`CreateContainer` 本身就已经带有可选的 `docker start`

5. 到这里，恢复顺序已经可以直接下最终工程序结论：

- `Virtual_Host_Agent.CreateContainer` 是主控函数
- 它先调用 `setStorage` 完成 `.img -> loop -> ext4 -> mount /data/android_data/<实例>`
- 然后执行 `docker create -v /data/android_data/<实例>:/data ...`
- 最后在 `IsStart` 条件满足时继续执行 `docker start <实例>`

因此当前异常已经几乎可以定性成：

宿主恢复链虽然代码顺序上确实是“先 mount，再 docker create，再 docker start”，但 live 结果仍落到了底层 `nvme0n1p6` 视图。这意味着问题更像不在“有没有先后顺序”，而在“setStorage 完成的那个 mount 对象，是否就是随后 docker create 所绑定到的同一个宿主 mount 对象 / 同一个传播域”。

换句话说，怀疑点已经从“是不是先 create 后 mount”进一步收缩成：

- `setStorage` 完成的 mount 是否落在 Docker create 可见的同一个 namespace / mount object 上
- `/data/android_data/<实例>` 在 `docker create` 执行瞬间，绑定到的是 loop overmount 还是底层被覆盖前目录
- 宿主上是否存在另一层 mount namespace / propagation 失配，导致虽然代码顺序正确，容器最终拿到的却不是 `setStorage` 刚挂好的那个 loop 视图

----

继续补了几个“足以改变判断边界”的关键验证后，需要把结论再收紧一次：

1. `LocalCommand` / `LocalCommandSU` 并没有做 `nsenter` 之类的 namespace 切换。

`Virtual_Host_Agent` 反编译显示：

- `LocalCommand` 本质上是 `sh -c <command>`
- `LocalCommandSU` 本质上是 `su -c <command>`

其中：

- `setStorage`
- `docker create ... -v /data/android_data/%s:/data ...`
- `docker start %s`

这些关键命令都走的是 `LocalCommand`，也就是普通 `sh -c` 执行路径，不是一个命令在宿主 namespace、另一个命令在某个 `nsenter`/容器 namespace 里执行。

所以“wrapper 本身切了 namespace，导致 mount 和 docker create 不在同一执行上下文”这个猜测，当前没有证据支持。

2. `dockerd`、`Virtual_Host_Agent`、`containerd-shim`、PID 1 当前都在同一个 mount namespace。

现场检查到：

- `dockerd`、`Virtual_Host_Agent`、`containerd-shim`、`init` 的 `ns/mnt` 都是同一个 `mnt:[4026531841]`
- 它们当前看到的 `/data/android_data/<74ae>` 也一致，都是宿主 `loop17` 那个 mount

这说明：

- 现在这一刻，并不存在 “agent 在一个 mount ns，dockerd 在另一个 mount ns” 的直接证据
- 所以之前那条“dockerd 在别的 namespace 里做 bind”的假设，当前需要降级，不再是最优先解释

3. 更关键的修正是：`74ae` 这次 08:30 后的恢复，并不是新建容器对象，而是旧对象被重新 start。

`docker inspect` 关键字段：

- `Created: 2026-05-07T11:59:11.888609494Z`
- `StartedAt: 2026-05-11T00:30:20.24237647Z`（对应北京时间约 08:30:20）
- `RestartCount: 0`

这说明：

- 这个容器对象本身 5 月 7 日就已经创建好了
- 5 月 11 日 08:30 只是旧对象被重新启动
- 所以当前异常实例并不是走“5 月 11 日重新 `docker create` 一个全新对象”的那条路径

这点非常重要，因为它意味着：

- `Virtual_Host_Agent.CreateContainer -> setStorage -> docker create -> docker start`

这条顺序虽然已经被我们坐实，但它更像“新建/创建实例”路径，不一定就是 74ae 这次 reboot 后恢复的真实主路径。

4. 对应旧对象恢复，AC_IaaS_agent 这边已经出现了另一条更贴近现场的调用链：

- `containerFunction.Restore`
- `pkg/containerUtils.ContainerStart`
- `pkg/dockersdk.(*DockerManager).StartContainer`

其中：

- `Restore` 在 `ContainerStart` 之前会做 `CheckContainerStatus`、`ContainerStop`、目录检查/创建、`DownloadFile`，以及至少两次 `fmt.Sprintf + execCommand`
- `ContainerStart` 则明确调用 `DockerManager.StartContainer`

这说明：

- 对 74ae 这种“旧容器对象 reboot 后恢复”的路径，主线更像是 AC_IaaS_agent 的 `Restore/ContainerStart`
- 而不是 Virtual_Host_Agent 的 `CreateContainer`

5. 因此到当前这一步，问题模型需要再修正成更贴近现场的一版：

- 新建实例路径：已经坐实是 `setStorage -> docker create -> 可选 docker start`
- 74ae 这次 reboot 恢复路径：更像是“旧对象 restore + start”，不是重新 create 新对象

所以现在最值得继续怀疑的，不再是 `CreateContainer` 里 `mount` 和 `docker create` 的先后顺序，而是：

- 旧容器对象在宿主 reboot 后被 `DockerManager.StartContainer` 启动时，它原先绑定的 `/data` source mount object 究竟引用的是哪一层对象
- 在旧对象 start 之前，宿主为 `/data/android_data/<实例>` 做的 remount / image restore 是否已经完成
- 旧对象的 bind source 是否在 container create 的历史时刻就已经钉到了底层目录，而 reboot 后只是再次沿用了这个旧绑定关系

换句话说，调查主战场又进一步收敛成：

- “新建路径顺序是否错误” 现在已经不是最优先怀疑点
- 更像是“旧容器对象的历史 bind source + reboot 后 restore/start 时机”这一类问题

----

继续把 reboot 当时的启动时间线补齐后，这个“旧对象恢复”模型又被进一步加强：

1. `74ae` 这个容器对象明确具备 Docker 自启动条件。

`docker inspect` 关键字段：

- `RestartPolicy: always`
- `MaximumRetryCount: 0`
- `Created: 2026-05-07T11:59:11.888609494Z`
- `StartedAt: 2026-05-11T00:30:20.24237647Z`（北京时间约 `08:30:20`）

这意味着宿主 reboot 后，即使没有 agent 显式再去 create，它本身也符合被 `dockerd` 自动重新拉起的条件。

2. boot 时间线也和“dockerd 自动恢复旧对象”高度一致：

- `PID 1 / init`: `08:30:08`
- `containerd`: `08:30:17`
- `dockerd`: `08:30:17`
- `runsvdir /etc/service`: `08:30:18`
- `AC_IaaS_agent`: `08:30:18`
- `Virtual_Host_Agent`: `08:30:18`
- 容器 `StartedAt`: `08:30:20`

这条时间线的关键信号是：

- `dockerd/containerd` 先于 agent 起来
- agent 和 dockerd 之间只差 1 秒
- 容器启动时间又紧跟在它们后面 2 秒左右

从系统行为上看，这非常符合“`dockerd` 在宿主 boot 后按 `restart=always` 自动恢复老容器对象”的形态。

3. 结合前面已经确认的事实：

- 这个对象不是 5 月 11 日新 create 的，而是 5 月 7 日老对象
- `dockerd` 与 agent 当前在同一个 mount namespace
- 新建路径里的 `setStorage -> docker create -> docker start` 虽然代码顺序正确，但不一定是 74ae 这次的真实路径

现在最强的现场解释已经进一步偏向：

宿主 reboot 后，`dockerd` 按 `restart=always` 自动去 start 这个 5 月 7 日创建的旧容器对象；而宿主侧针对 `/data/android_data/<实例>` 的 image remount / restore 动作，与这个 auto-start 的时机没有完全对齐，于是旧对象在 start 时拿到的仍是底层目录对象，之后宿主再把同一路径 overmount 成 loop 视图，最终形成“宿主看到 loop17，live 容器看到 nvme0n1p6 子目录”的分叉。

4. 这会把当前主怀疑点再收缩一层：

- 不再优先怀疑 `Virtual_Host_Agent.CreateContainer` 里的 `docker create` 路径
- 更优先怀疑 `dockerd` 的老对象 auto-start 与宿主 remount `/data/android_data/<实例>` 之间的竞争窗口

换句话说，最像的机制已经从“agent 内部命令顺序错误”进一步收敛成：

宿主 reboot 后，老容器对象被 `dockerd` 依据 `restart=always` 很早自动拉起，而 `/data/android_data/<实例>` 对应的 loop 镜像恢复/重挂尚未稳定完成，导致旧对象 start 时绑定到的是底层目录对象；之后宿主 overmount 成 loop17，只改变了宿主视图，没有改掉 live 容器已持有的 `/data` 挂载结果。

----

继续把 AC_IaaS_agent 里的“业务 restore”与“真正挂载 helper”拆开之后，又能再排掉一条误导分支：

1. `containerFunction.restore` 里那两次 `fmt.Sprintf + execCommand`，现在看起来更像业务恢复/文件落盘逻辑，不像 boot 时的数据面 remount。

静态还原到的命令模板里，至少有：

- `tar -xf %s -C %s`
- `mv %s %s`

而不是我们之前更关心的 loop mount / bind mount 命令。

2. AC 里真正和 loop mount 强相关的 helper 其实是另一组函数：

- `containerFunction.mountDataImg`
- `imgService.Mount`
- `imgService.mount`
- `pkg/utils.UmountLazy`

这些函数对应的精确命令字符串已经能看到：

- `mount -o loop %s %s`
- 以及配套的 `umount` / `resize2fs` / `e2fsck` / `losetup` 一类字符串

3. 更关键的是，`mountDataImg` 的调用者并不是 `Restore`，而是迁移/回滚相关路径：

当前定位到的三个调用点都落在：

- `containerFunction.RollbackMigration`
- `containerFunction.migrationMount`
- `containerFunction.migrationRestore`

也就是说：

- `containerFunction.restore` 负责的是“业务数据 restore / 解包 / 落盘 / 再 start”这一类逻辑
- `mountDataImg` 才是 AC 里真正做 loop image mount 的 helper
- 但 `74ae` 这次 reboot 恢复线，当前没有证据表明它走到了这些 migration/mountDataImg 路径

4. 这会再把调查主线收紧一层：

- 不应再把 `containerFunction.restore` 当成宿主 reboot 时为 `/data/android_data/<实例>` remount loop 镜像的主逻辑
- 也不应把它和 `Virtual_Host_Agent.CreateContainer` 那条“新建实例”路径混为一谈

到当前为止，更贴近现场的模型已经变成：

- `74ae` 是 5 月 7 日创建的旧容器对象
- 宿主 5 月 11 日 reboot 后，`dockerd` 依据 `restart=always` 很早自动去 start 这个旧对象
- AC 的 `restore` 更像业务 restore，而非 boot remount
- AC 里真正做 loop mount 的 `mountDataImg` 目前只在 migration/rollback 一类路径上看到调用

因此当前最强判断是：

这次异常更像是 Docker 对旧容器对象的 auto-start 与宿主底层 `/data/android_data/<实例>` 挂载恢复之间发生了竞争，而不是某个显式的 AC/Host_Agent “restore”函数先后顺序写错。

换句话说，现在主战场已经进一步压缩到：

- `dockerd` 对旧对象的 auto-start 时机
- 宿主在 boot 初期何时把 `/data/android_data/<实例>` 切回 loop 镜像
- 两者之间是否存在短暂窗口，让旧对象 bind 到了底层目录对象并沿用到整个运行期

----

最后再补一个很有区分度的调用关系，可以把 Host_Agent 这边彻底收口：

1. `Virtual_Host_Agent.setStorage` 当前只找到一个调用者，而且就是 `VirtualHostAgent/virtualization.CreateContainer`。

现场静态结果：

- `CALL VirtualHostAgent/virtualization.setStorage` 只出现 1 次
- 调用点地址正落在 `VirtualHostAgent/virtualization.CreateContainer` 内部

没有看到：

- `RebootContainer`
- `StartCon`
- `StopCon`
- 其他 host-side restart/query 路径

对 `setStorage` 的复用调用。

2. 这条关系非常关键，因为它意味着：

- `setStorage` 是“新建容器”路径专属的 storage setup helper
- 它不是一个在 reboot 后对旧对象普遍复用的宿主 remount 入口

换句话说，如果 `74ae` 这次在 2026-05-11 08:30 只是“旧对象 auto-start”，那么当前没有证据表明它会再次走到 `Virtual_Host_Agent.setStorage`。

3. 结合前面已经坐实的事实：

- `74ae` 是 5 月 7 日创建的旧对象
- reboot 后它符合 `restart=always` 自动拉起条件
- `dockerd/containerd` 比 agent 更早起来
- `setStorage` 只属于 `CreateContainer` 新建路径

5. 这条证据不能证明“容器内 bind mount 建立晚于 init”，因为 OCI/runc 本来就是先准备容器 mounts，再 exec PID 1；但它确实开始支持另一件更关键的事：

宿主对 `/data/DockerMount/<实例>.img -> /data/android_data/<实例>` 这一层最终 loop/ext4 挂载完成的时间，并不早于该旧对象容器 init 的启动时间，至少已经出现了“init 在 `08:30:19`，镜像最近 mount 记录在 `08:30:20`”这一秒级倒挂。

6. 因为 `ps lstart` 与 ext4 `Last mount time` 都只有秒级精度，所以这里还不能把结论写成绝对严格的“精确先后铁证”；但和之前相比，判断已经从：

- “只是机制上怀疑存在 auto-start vs remount 竞争”

推进到：

- “在 `74ae` 同实例时间戳上，已经看到容器 init (`08:30:19`) 早于或至少不晚于镜像最终 mount 记录 (`08:30:20`) 的直接信号”

7. 这会让当前主模型再收紧一步：

- `74ae` 旧对象在宿主 reboot 后很早被拉起
- 容器 init 已经在跑时，宿主这边对应 `.img` 的最终 ext4/loop mount 记录才落在下一秒
- 这与前面 live mountinfo 看到的结果完全同向：容器 `/data` 固定在底层 `nvme0n1p6` 对象，而宿主后来看到的是 `loop17`

因此到这一步，关于“是不是容器先活了、宿主后把镜像挂上去”这件事，已经不只是抽象机制推演，而是有了 `74ae` 同实例时间戳层面的正向支持。
